# Spec Owner Kit

> AI(Claude)가 **Spec Owner 역할**을 수행할 수 있도록 설계된 프로젝트 템플릿

회의록, 요구사항 등 원본 자료를 넣으면 AI가 자동으로 **기능정의서, 화면정의서, WBS** 등 산출물을 생성합니다.

## 특징

- **AI 친화적 구조**: `inputs/` → AI 분석 → `outputs/` 자동 생성
- **Pre-flight 체크리스트**: AI가 작업 전 파일 인벤토리 확인
- **Spec Owner 프로세스 내장**: 스캔 → 분석 → 정의 → WBS → Freeze 5단계
- **다양한 파일 형식 지원**: `.md`, `.docx`, `.txt`, `.pdf`, `.png` 등
- **미확정 항목 관리**: 협의 필요 사항 체계적 추적
- **핸드오프 체크리스트**: 역할별(UI Composer, Integrator, Verifier) 전달 사항
- **서브에이전트 내장**: 6개 전문 에이전트가 자동 로드되어 작업 지원

## 서브에이전트

> Clone 시 자동으로 로드되는 전문 AI 에이전트들

| 에이전트 | 역할 | 자동 호출 조건 |
|---------|------|---------------|
| **lukuku-spec-owner** | 기술 스펙 전문가 | 스펙 문서 작성, 요구사항 정의 요청 시 |
| **scope-guard** | 스코프 검증 | 요구사항 검토, 범위 검증 요청 시 |
| **shopify-schema-architect** | Shopify 데이터 설계 | Metafield/Metaobject 설계 요청 시 |
| **edge-case-hunter** | 엣지케이스 분석 | 기능 구현 후 실패 시나리오 분석 시 |
| **wbs-orchestrator** | WBS 설계 | 스펙 확정 후 일정/태스크 분할 시 |
| **ac-standardizer** | AC 표준화 | 모호한 요구사항을 측정 가능한 AC로 변환 시 |

### 사용 방법

**자동 호출**: 작업 내용에 맞는 에이전트가 자동으로 선택됩니다
```
"이 요구사항 검토해줘"  → scope-guard 자동 호출
"엣지케이스 분석해줘"  → edge-case-hunter 자동 호출
```

**명시적 호출**: 특정 에이전트를 직접 지정할 수도 있습니다
```
"scope-guard 에이전트로 이 기능 분석해줘"
"wbs-orchestrator로 일정 짜줘"
```

```
│   └── agents/            # ⭐ 서브에이전트 (자동 로드)

│       ├── scope-guard.md
│       ├── shopify-schema-architect.md
│       ├── edge-case-hunter.md

│       └── ac-standardizer.md
│
├── inputs/                # 원본 자료 (여기에 자료 넣기)

│

```

## 시작하기

### 1. 클론
```bash
git clone https://github.com/ej-rarus/spec-owner-kit.git 프로젝트명
cd 프로젝트명
```

### 2. 프로젝트 정보 수정
`CLAUDE.md` 파일에서 플레이스홀더 수정:
```markdown
- **프로젝트명**: {{PROJECT_NAME}} → 실제 프로젝트명
- **클라이언트**: {{CLIENT_NAME}} → 실제 클라이언트명
- **기간**: {{START_DATE}} ~ {{END_DATE}}
```

### 3. 자료 넣기
```
inputs/회의록/      ← 미팅 기록 (.docx, .txt, .md 등)
inputs/요구사항/    ← 고객 요청사항
inputs/참고자료/    ← 벤치마킹, 스크린샷 (.png, .jpg 등)
```

**지원 파일 형식**:
| 형식 | 처리 방법 |
|------|----------|
| `.md` | 직접 읽기 |
| `.txt` | 직접 읽기 (대용량 시 분할) |
| `.docx` | macOS에서 자동 변환 |
| `.pdf` | 직접 읽기 |
| `.png/.jpg` | 이미지 분석 |

### 4. AI에게 요청
```
"inputs 폴더 스캔해서 INDEX.md 만들어줘"   ← Phase 0 (권장)
"inputs 자료 읽고 기능정의서 만들어줘"
```

## 작업 프로세스

```
Phase 0          Phase 1          Phase 2          Phase 3          Phase 4
[스캔]    →     [분석]    →     [정의]    →     [WBS]    →     [Freeze]
   │               │               │               │               │
   ▼               ▼               ▼               ▼               ▼
파일 인벤토리   회의록 분석     기능정의서      WBS 작성      핸드오프
INDEX.md 생성  요구사항 정리   화면정의서      일정 확정     Spec 고정
⭐사용자 확인   미확정 도출    AC/Edge Case    리스크 관리   팀 공유
              ⭐사용자 확인   ⭐사용자 확인
```

### Phase 0: 스캔 (NEW)
AI가 작업 전 반드시 수행하는 파일 인벤토리 확인:
- 모든 파일 형식 확인 (.md뿐 아니라 .docx, .txt, .png 등)
- 대용량 파일 식별 (50KB 이상)
- `inputs/INDEX.md` 생성 또는 업데이트
- 사용자에게 파일 목록 확인 요청

## AI 명령어

|--------|------|

| 명령어 | 용도 |

| `"각 기능별 Edge Case 도출해줘"` | 예외상황 도출 |

### Phase 3: WBS
| 명령어 | 용도 |
|--------|------|

| `"리스크 항목 식별해줘"` | 리스크 도출 |

### Phase 4: Freeze
| 명령어 | 용도 |
|--------|------|
| `"미확정 항목 중 아직 해소 안된 거 있어?"` | 미확정 점검 |

## Spec Owner 역할

> **Spec Owner**는 제품의 **논리적 무결성**과 프로젝트의 **적기 인도**를 책임지는 역할

| 책임 | 설명 |
|------|------|
| 범위 확정 | In/Out of Scope 정의 |
| 비즈니스 로직 | 정책 및 규칙 확정 |
| WBS 관리 | 일정 수립 및 모니터링 |
| 핸드오프 | 개발팀에 명세 전달 |

## 주의사항

### 샘플 파일 처리
- `_samples/` 폴더의 파일은 **참고용 샘플**입니다
- AI 작업 시 자동으로 제외됩니다
- 실제 프로젝트 자료는 반드시 `inputs/` 폴더에 넣으세요

### 대용량 파일
- 50KB 이상 파일은 분할 읽기가 필요할 수 있습니다
- AI가 자동으로 감지하고 처리합니다

### 단계별 확인
- 각 Phase 완료 후 AI가 사용자 확인을 요청합니다
- 중간 확인 없이 전체 작업을 한 번에 진행하지 않습니다

## 팁

1. **Phase 0 먼저**: 작업 시작 전 `INDEX.md` 생성 권장
2. **자료를 충분히 넣기**: inputs에 자료가 많을수록 AI 결과물 품질 향상
3. **단계별 진행**: Phase 0 → 1 → 2 → 3 → 4 순서 준수
4. **미확정은 바로 기록**: 모호한 부분 발견 즉시 미확정항목에 추가
5. **일관성 유지**: 용어, 화면ID, 기능명 통일
